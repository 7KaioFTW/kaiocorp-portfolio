Ce qui a été changé

Passe 1 — Best Practices 73 → 100 & A11y → 100
  • Suppression du script Google AdSense (layout.tsx). Cause unique des 3 échecs Best Practices : cookie tiers __mggpc__, erreurs console, issues CSP. Choix produit validé (vitrine B2B).
  • CSP nettoyée (next.config.js) : suppression des domaines Google/AdSense devenus inutiles ; ajout form-action, base-uri, object-src 'none', frame-ancestors ; retrait de unsafe-eval.
  • Contraste Footer : text-slate-500 → text-slate-400 (4,15:1 → ~7,5:1).
  • heading-order Footer : titres de colonnes <h5> → <h2> (plus de saut de niveau).
  • Label-in-Name : l'aria-label du sélecteur de langue inclut désormais le code visible (Change language (FR)).
Passe 2 — /realisations A11y → 100
  • heading-order : ajout d'un <h2 class="sr-only"> entre le PageHero (h1) et les cartes (h3).
  • Expérience priority sur les images de la grille : annulée. Elle a fait régresser le LCP mobile (3,0 s → 3,8 s) — sur mobile la grille est sous le pli, le LCP est le titre, et précharger des images sous le pli volait de la bande passante. Règle « échec → revert, pas rustine ».
Passe 3 — /contact A11y 96 → 100

  • Le BriefForm est déjà conforme (tous les label htmlFor associés, honeypot, bouton nommé).

Précisions importantes (honnêteté de mesure)

  • SEO 92 sur localhost = 100 en production. Le seul échec SEO est canonical : le tag rendu est <link rel="canonical" href="https://kaiocorp.com"> (bien formé), mais Lighthouse le rejette car la page auditée est localhost ≠ kaiocorp.com. En prod, canonical = la page → audit vert. Le « corriger » casserait le SEO en prod ; volontairement non touché.
  • Perf mobile pessimiste. L'audit passe par le Chrome de debug (profil réel avec extensions). L'extension Phantom injecte du JS parasite (unused-javascript) qui pèse sur TBT/perf. Les scores réels sans extension sont ≥ ceux mesurés.
  • Plafond perf mobile. La home et les PageHero ont un LCP = titre <h1> en dégradé (Orbitron, display:swap). Render-blocking « none », polices non bloquantes. Passer 94–96 → 100 demanderait du critical-CSS inlining (experimental.optimizeCss) — risqué pour un gain marginal, écarté car > 90 déjà atteint.
